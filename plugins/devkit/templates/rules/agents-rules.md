プロジェクト固有ルールは、このセクションの外側に書く。

## DevKit Workflow

開発フローは `/dig` を前提に進める。

1. 深掘り: 要求が曖昧な場合は未知を棚卸しし、影響が大きい未知だけ質問する。小さい未知は仮定を明示して進める。
2. 計画: コードベースを調査し、目的、成功条件、非対象、write_scope、検証方法が揃った計画を作る。
3. 承認: 実装前に計画を提示し、ユーザー承認を得る。
4. 実装: 計画と write_scope に沿って差分を作る。逸脱が必要な場合は理由、リスク、要確認点を記録する。
5. 自レビュー: diff 全文を計画と突き合わせ、必要なテスト、リンタ、検証コマンドを実行する。
6. 修正ループ: 指摘や検証失敗があれば修正し、指摘が解消するまで再確認する。
7. 報告: 変更内容、検証結果、残課題、計画からの逸脱を簡潔に報告する。

## Review Rules

- ファイル変更を伴うタスクでは、独立した review を 1 回以上実施する。
- review で指摘が出た場合は修正後に再 review を行い、追加 findings がなくなるまで繰り返す。
- 品質ルールは prose より決定論的な check を優先し、同じ失敗を次回検出できる形に寄せる。

## Commit Rules

- コミットメッセージは Conventional Commits を使う。
- 基本形は `<type>(<scope>): <summary>` とし、`scope` は必要な場合だけ付ける。
- `type` は `feat` `fix` `docs` `refactor` `test` `chore` `ci` `build` `perf` `revert` を優先して使う。
- `summary` は日本語で簡潔に書く。
- 本文を書く場合も日本語で統一し、変更理由、影響範囲、補足を必要最小限で書く。
- 破壊的変更は `type!:` または `type(scope)!:` を使い、必要なら本文に `BREAKING CHANGE:` で日本語説明を付ける。
- commit / push はユーザーが明示した場合のみ行う。
