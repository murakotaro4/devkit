# /goal-prompt - ゴールプロンプト作成

親エージェント = インタビュー・read-only 調査・ゴールプロンプト組み立て・起動コマンド推奨・確認済みの任意投入。自律・長時間実行 / 定期ループ実行 / 大タスク完走依頼で、停止条件と成功条件が曖昧なまま走り出す失敗を防ぐ。

## 対象

$ARGUMENTS

## ハーネス判定

正本は devkit リポジトリの `AGENTS.md`「スキル共通契約」。この SKILL.md は Claude 親 / Codex 親の二層構成で実行する。要点:

- `AskUserQuestion` が使える -> Claude 親。
- なければ `spawn_agent` が使える -> Codex 親。
- どちらでもない -> 判定不能として扱い、選択肢を箇条書きで提示して自由文回答を求める。
- `request_user_input` は plan mode 依存で不安定なため、判定キーに使わない(Codex 親 plan mode の質問手段としてのみ使う)。

## タスクリスト連動

正本は devkit リポジトリの `AGENTS.md`「スキル共通契約 > タスクリスト連動」。goal-prompt 開始時に step 1-9 をタスクリストへ登録し、各 step の開始時に `in_progress`、完了時に `completed` へ更新する(Claude 親: TaskCreate / TaskUpdate、利用不可なら省略可。Codex 親: 組み込み plan 機能または通常の進捗報告で同等の進捗提示)。

## 書き込み契約

- step 1-7 は read-only。Bash は `command -v`、`git status`、`rg`、`ls`、`find`、テスト・lint コマンドの存在確認など読み取り用途だけに使う。Bash での書き込み、mkdir、touch、ファイル編集は行わない。
- step 8 は、ユーザーがファイル保存を選択した場合に Write で新規保存できる。既定パスは `docs/goals/YYYY-MM-DD-<slug>.md`。保存前にパスを確認し、同名ファイルは上書きせず `-2` などの連番にする。
- step 9 は、ユーザーが実行を選択した場合のみ実行する。実行できる経路は codex exec 投入、`loop` スキル起動、セッション内引き継ぎに限る。cron 登録、`/schedule` 登録、commit、push はどの step でも行わない。
- 保存前後を問わず、秘密情報、資格情報、トークン、個人情報をプロンプト本文へ転記しない。必要な場合は「環境に既に存在する設定を読む」と書き、値そのものは書かない。

## 実行メカニズム一覧

原則は「長い本文はファイル参照、CLI 引数と条件文には短い指示だけ」。選択した機構の長さ・受け渡し制約に合う形へ組み立てる。

| 場面 | 機構 | スキルが実行可 / 提示のみ | 長さ・受け渡し制約 |
|------|------|---------------------------|--------------------|
| セッション内で条件達成までターン継続 | Claude Code `/goal` | 提示のみ。ユーザータイプ専用 | `/goal` 条件文 4,000 字上限。渡すのは検証可能な条件 + 停止バウンドだけにし、本文はファイル参照 |
| 開発中のポーリング・巡回 | `/loop` または `loop` スキル | Claude 親で `loop` スキルが存在する場合のみ実行可。不在時は提示のみ | 短い指示にし、長い本文はファイル参照 |
| 夜間・定期・マシン非依存 | `/schedule` | 提示のみ。登録はしない | 最小 1h。1h 未満の間隔なら `/loop` を推奨 |
| ワンショット自律実行 | headless `claude -p` | 提示のみ | 非公式の入れ子実行として扱い、保存ファイル参照の短い起動指示だけを提示 |
| Codex 側の自律実行 | `codex exec` | Claude 親のみ実行可。Codex 親は提示のみ | `codex -a never exec -C "<repo>" --sandbox workspace-write -c model_reasoning_effort="<effort>" "<保存ファイルパス> を読み、その指示に従って作業せよ" < /dev/null`。`"$(cat file)"` 展開は使わない |
| 大タスク完走 | 通常セッション貼付け / 引き継ぎ | 提示のみ。実装系は `$dig` 推奨 | 現在セッションの権限、plan mode、repo ルールに従って親エージェントへ引き継ぐ |

step 2 の事前調査で `command -v claude` と `command -v codex` を確認し、CLI 不在なら該当コマンドは提示しない。slash command は CLI から存在確認できないため「推奨（ハーネスのバージョンにより存在しない場合あり）」と注記し、fallback として通常セッション貼付けまたは headless 実行を 1 つ添える。

## 排除する失敗モード

1. 停止条件欠落: 全プロンプトに達成停止、上限停止、行き詰まり停止の 3 種を必須化する。上限停止は省略禁止。
2. ゴール誤解釈・スコープドリフト: 成功条件は機械検証(コマンド + exit code)を第一候補にする。検証コマンドがない調査・文書化系は「成果物の存在 + 客観的な手動確認手順(チェックリスト)」へ fallback する。エージェント自己申告のみの成功条件は禁止。
3. 変更範囲の膨張: 非対象、変更禁止領域、破壊的操作の扱いを本文へ明記する。
4. 無限待機: 「実行中は質問不可。曖昧な点は保守的解釈 + 判断を進捗ログに記録」と本文に焼き込む。
5. 受け渡し失敗: 長い本文はファイル参照にし、CLI 引数と `/goal` 条件文には短い指示だけを渡す。

## 質問表

1 ラウンド最大 4 問、必ず選択肢付きで聞く。親の推奨を付け、影響が小さい未知は仮定を明示して進める。Codex 親 plan mode で `request_user_input` を使う場合は 1 呼び出し最大 3 問に分ける。

| ラウンド | 目的 | 必ず聞くこと |
|----------|------|--------------|
| Round 1: 実行形態と対象 | どの機構へ渡すプロンプトかを決める | 形態(`/goal` / `/loop` / `/schedule` / headless / codex exec / 通常セッション)、ハーネス(Claude 親 / Codex 親 / 判定不能)、対象 repo または対象ファイル、タスク型(実装 / 調査 / 監視 / 文書化 / 整理) |
| Round 2: ゴール定義 | 終了状態を検証可能にする | 完了状態、検証方法、検証コマンド、品質バー |
| Round 3: 境界と停止条件 | 暴走・破壊・迷走を防ぐ | 非対象、上限停止の種類と値、行き詰まり時の扱い、破壊的操作の可否 |
| Round 4: 運用 | 長時間実行の観測性と再開性を決める | 進捗の残し方、権限、途中中断と再開、完了報告の形式 |

## プロンプトテンプレート

生成するゴールプロンプトは次の節をこの順序で持つ。

````markdown
# Goal Prompt: <短い名前>

## 目的
<何を達成するか。背景は必要最小限>

## 成功条件(検証可能)
- <機械検証できる条件、または成果物 + 客観的な手動確認手順>

## 検証コマンド
```bash
<実行する検証コマンド。ない場合は「なし」とせず手動確認手順を書く>
```

## 制約・非対象
- <触らないファイル、やらない作業、権限の境界>
- <破壊的操作の扱い>

## 停止条件(3 種)
- 達成停止: <成功条件を満たしたら停止>
- 上限停止: <最大ターン数、最大時間、最大ループ回数など。省略禁止>
- 行き詰まり停止: <同じ blocker が続く、外部入力が必要、検証不能など>

## 進捗報告
- <ログ場所、報告頻度、増分の書き方>

## 実行前提
- 実行中は質問不可。曖昧な点は保守的解釈 + 判断を進捗ログに記録する。
- 秘密情報は本文に含めない。
````

## セルフチェック

step 6 で、提示前に 6 項目を必ず確認する。

1. 検証可能性: 成功条件がコマンド + exit code、または成果物 + 客観的な手動確認手順になっている。
2. 上限停止必須: 達成停止、上限停止、行き詰まり停止の 3 種があり、上限停止は省略禁止になっている。
3. 非対象・破壊的操作明記: 触らない範囲と破壊的操作の可否が明記されている。
4. 実行形態との無矛盾: `/goal`、`/loop`、`/schedule`、headless、codex exec、通常セッションの制約と矛盾しない。
5. 秘密情報なし: 値そのものではなく参照方法だけを書いている。
6. 機構の長さ・受け渡し制約への適合: `/goal` 条件文 4,000 字上限、長い本文はファイル参照、CLI 引数は短い起動指示だけ、という原則を満たしている。

## 起動コマンド決定表

| 形態 | ハーネス | 推奨出力 | スキルが実行可 / 提示のみ |
|------|----------|----------|---------------------------|
| `/goal` | Claude 親 | `/goal docs/goals/<file>.md の成功条件をすべて満たす or stop after N turns` | 提示のみ |
| `/loop` | Claude 親 | `/loop 5m docs/goals/<file>.md を読み、進捗ログを更新しながら条件を満たすまで確認する` | `loop` スキルが存在する場合のみ実行可。不在時は提示のみ |
| `/schedule` | Claude 親 | `/schedule <cron or hourly trigger> docs/goals/<file>.md を読む` | 提示のみ。登録はしない |
| headless | Claude 親 | `claude -p "docs/goals/<file>.md を読み、その指示に従って作業せよ" --permission-mode acceptEdits --allowedTools "..."` | 提示のみ |
| codex exec | Claude 親 | `codex -a never exec -C "<repo>" --sandbox workspace-write -c model_reasoning_effort="<effort>" "docs/goals/<file>.md を読み、その指示に従って作業せよ" < /dev/null` | 実行可 |
| codex exec | Codex 親 | `codex -a never exec -C "<repo>" --sandbox workspace-write -c model_reasoning_effort="<effort>" "docs/goals/<file>.md を読み、その指示に従って作業せよ" < /dev/null` | 提示のみ |
| 通常セッション | Claude 親 / Codex 親 | 生成したプロンプトを現在セッションの通常ルールへ引き継ぐ。実装系タスクで `$dig` が利用可能なら `$dig` 推奨 | 提示のみ |

## フロー

### 1. Round 1: 実行形態と対象

選択肢付き質問で、形態、ハーネス、対象 repo / ファイル、タスク型を決める。締めに目的 / 成功条件の仮説 / 非対象の仮説 / 採用した仮定を短く提示する。

### 2. 事前調査 read-only

対象 repo やファイルを read-only で確認する。`command -v claude`、`command -v codex`、テスト・lint・format コマンド、既存 docs、実行ログの置き場所を調べ、起動コマンド候補と検証コマンド候補を具体化する。

### 3. Round 2: ゴール定義

完了状態、検証方法、検証コマンド、品質バーを聞く。選択肢は step 2 の調査結果から具体化し、機械検証できない場合は成果物 + 手動確認チェックリストへ fallback する。

### 4. Round 3: 境界と停止条件

非対象、上限停止の種類と値、行き詰まり時の扱い、破壊的操作の可否を聞く。上限停止は省略禁止として扱い、未指定なら親の推奨値を提示して確認する。

### 5. Round 4: 運用

進捗の残し方、権限、途中中断と再開、完了報告の形式を聞く。長時間実行ではログ場所、報告頻度、再開時に読むファイルを明記する。

### 6. プロンプト組み立て + セルフチェック

プロンプトテンプレートへ回答を統合し、セルフチェック 6 項目を満たすまで修正する。長い本文はファイル参照へ寄せ、`/goal` 条件文や CLI 引数を短く保つ。

### 7. 最終確認

完成プロンプト全文と推奨起動コマンドを提示し、修正が必要か確認する。修正があれば step 6 に戻る。実行はここではまだ行わない。

### 8. 出力と起動コマンド提示

ユーザーが保存を選んだ場合、既定パス `docs/goals/YYYY-MM-DD-<slug>.md` を提示して確認し、Write で新規保存する。headless、codex exec、`/goal`、`/loop` などファイル参照が必要な機構では保存を推奨する。保存しない場合は、通常セッション貼付け用の本文だけを提示する。

### 9. 実行(オプション、確認制)

必ず「実行しますか？」と確認してから進める。Codex 親は常に生成 + コマンド提示までで、入れ子実行しない。

Claude 親で codex exec 投入を選んだ場合、投入できるのはセルフチェック 6 項目を通過し、制約・非対象 / 破壊的操作 / 停止条件の節が埋まった保存済みファイルのみ。固定形は次のとおり。

```bash
JOB_DIR=$(mktemp -d "${TMPDIR:-/tmp}/devkit-goal-job.XXXXXX") && echo "JOB_DIR=$JOB_DIR"
codex -a never exec -C "<対象repo>" --sandbox workspace-write -c model_reasoning_effort="<effort>" "<保存したゴールプロンプトファイルの絶対パス> を読み、その指示に従って作業せよ" < /dev/null > "$JOB_DIR/codex.log" 2>&1 &
```

echo された `JOB_DIR` のパスを親が記録する。シェル変数はツール呼び出し間で失われるため、以降の監視・引き渡し・ログ案内では記録済みパスを使う。

read-only 系タスクは `--sandbox read-only` を使う。投入時に「監視」または「引き渡し」を 1 問で選ぶ。監視は 60-120 秒ハートビートで、時刻・経過・ログ増分、完了報告、停止条件が機能したかの確認を含める。引き渡しは記録済み `JOB_DIR` のログパスと確認方法を案内して終了する。

定期ループを選んだ場合、利用可能スキル一覧に `loop` が存在する場合のみ `Skill(skill: "loop", args: "<interval> <短い指示（本文はファイル参照）>")` で起動する。`loop` スキル不在時の fallback は `/loop` コマンド提示。ループはセッションが開いている間だけ動く前提を必ず注記する。

大タスク完走を選んだ場合、goal-prompt の書き込み契約はスキル終了とともに終わり、生成済みプロンプトを親エージェントが通常セッションのルール(ユーザーの権限設定、plan mode、repo ルール)の下で実行する。実装系タスクで `$dig` が利用可能なら `$dig` への引き継ぎを推奨する。

`/goal`、`/schedule`、headless `claude -p` はコマンド提示のみ。`/goal` はユーザータイプ専用、`/schedule` は登録しない、入れ子 `claude -p` は非公式のためスキルから投入しない。

## 注意

- 裸の `/goal` をトリガーとして扱わない。組み込み `/goal` との誤発火を避ける。
- `codex -a never exec` のコマンド例を書く場合は、非対話実行でハングしないよう必ず同じ行に `< /dev/null` を含める。
- codex コマンド例に `-m` でモデルを焼き込まない。モデルは config 既定に従い、調整するのは `model_reasoning_effort` だけにする。
- cron 登録、`/schedule` 登録、commit、push は行わない。
- 実行可否や CLI の存在が不明な場合は提示のみへ落とし、fallback を添える。
