# /goal-prompt - Goal プロンプト保存生成

実装スキルではない。既にある程度固まった仕様・会話・レビュー済みの計画を、意味を変えずに Goal 実行形式へ変換して `.claude/goal-runs/` へ保存し、最後にユーザーがそのまま送信できる起動プロンプトを出す。

## 対象

$ARGUMENTS

## 対象とする依頼

- この会話から Goal プロンプトを作って
- さっきの計画を /goal で実行できる形にして
- この Issue の内容を Goal プロンプトにして保存して
- この仕様書から Goal 実行プロンプトを作って
- 後で Claude Code で実行する起動プロンプトを作って

## ハーネス判定と質問手段

正本は devkit リポジトリの `AGENTS.md`「スキル共通契約」。要点:

- `AskUserQuestion` が使える -> Claude 親(AskUserQuestion で質問)。
- なければ `spawn_agent` が使える -> Codex 親。
- どちらでもない -> 判定不能として扱う。
- Codex 親 plan mode の質問手段は `request_user_input`。Codex 親通常 mode / 判定不能は、選択肢を箇条書きで提示して自由文回答を求める。

## 入力ソース解決

優先順位は次のとおり。

1. ユーザーが明示したファイル・Issue・PR・仕様書
2. dig が保存した `.claude/plans/*.md`
3. 現在セッションの会話内容
4. ユーザーが自然文で渡した仕様

ユーザーが情報源を明示した場合は、常にその指定を最優先で使う。具体的なファイル・URL・パスの指定(`$ARGUMENTS` 含む)は 1 として扱い、「この会話から」は 3、「この仕様から」等の自然文指定は 4 を直接指す明示として扱う(明示された情報源より 2 の保存済み計画を優先しない)。2 → 3 → 4 の探索順は、ユーザーが情報源を明示しなかった場合だけ適用する。`.claude/plans/*.md` を採用するときは選んだ計画ファイル名を最終出力に明示し、複数の計画が候補になり判断できない場合は短く確認する。

## 不足情報の扱い

目的・成功条件・変更範囲・非対象・検証方法・完了証拠を入力ソースから抽出する。重大な不足(成功条件が立てられない、write_scope が判断できない等)がある場合だけ 1 ラウンド以内で短く確認する。軽微な不足は安全側の既定値で補い、採用した既定値を Goal 本文と最終出力の両方に明示する。

## 上限停止の自動算出

毎回ユーザーへ聞かない。ユーザーが明示した場合だけその値で上書きする。既定値は次のとおり自動算出する。

| 規模 | 上限停止 | 修正ループ | CI 待機 |
|------|----------|------------|---------|
| 小規模 | 12 ターンまたは 45 分 | 2 周 | 20 分 |
| 標準 | 24 ターンまたは 120 分 | 3 周 | 30 分 |
| 大規模 | 40 ターンまたは 240 分 | 4 周 | 45 分 |

大規模を超える場合は上限を無理に伸ばさず、複数 Goal への分割を提案する。

## Goal プロンプト本文の必須項目

生成する Goal 本文には次の項目をすべて含める。

- 実行モード: 不在自律実行(質問・確認・承認求めを出力して停止しない)
- 目的
- 成功条件(検証可能)
- 検証コマンドまたは客観的な確認手順
- write_scope または read_scope
- 非対象
- 破壊的操作・外部状態変更の扱い
- 上限停止(省略禁止。上記の自動算出値または明示上書き値)
- 行き詰まり停止
- 実装後に会話へ提示する完了証拠
- 完了レポート保存先(`.claude/goal-runs/` 内)
- 秘密情報を本文に含めない指示(値そのものではなく参照方法だけを書く)

## 保存契約

保存先は対象リポジトリまたは現在の作業ディレクトリ基準で `.claude/goal-runs/YYYY-MM-DD-<slug>-goal.md`。既存ファイルがある場合は上書きせず、`-goal` サフィックスの前に連番を挟んだ `YYYY-MM-DD-<slug>-2-goal.md` から採番する。

`.claude/goal-runs/.gitignore` が無ければ `*` 1 行で新規作成し、既存の `.gitignore` は内容を変更しない。git 管理下では保存後に `git check-ignore` で保存ファイルが ignore されているか確認し、ignore されていなければ最終出力で警告する。非 git ディレクトリでは検証不能のため skip する。

## 起動プロンプト出力

保存後、最後に次の基本形を出力する。

```
/goal .claude/goal-runs/YYYY-MM-DD-<slug>-goal.md を読み、記載された成功条件を満たすまで実行する。完了時は成功条件ごとの達成状況、検証コマンドと終了コード、変更ファイル、残課題を会話へ提示する。上限停止: <自動算出した値>。
```

標準は repo 相対パスを使う。必要に応じて絶対パス版を併記してよい。Codex や Cursor 向けの出力は、ユーザーが明示した場合だけ、同じ Goal 本文を読む自己完結プロンプトを出す(Claude Code の `/goal` と同じ継続制御があるとは書かない)。

## 禁止事項

- コード変更しない。
- 実装しない。
- PR を作らない。
- commit しない。
- push しない。
- 計画レビューしない。
- Goal プロンプトの独立レビューを行わない(レビュー済みの計画が正本。意味を変えずに変換するだけ)。
- Claude Code 組み込み `/goal` を自動発動しない(`/goal` の実行はユーザーが行う)。
- scheduler・loop 登録をしない。
- thought-db へ書き込まない。

Goal プロンプト保存ファイルと `.claude/goal-runs/.gitignore` の作成だけが、このスキルの書き込みである。
