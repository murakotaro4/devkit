# /refactor - 技術的負債の棚卸し + dig 引き継ぎ

親エージェント = スコープ確認・read-only 調査・優先順位付け・リファクタリング計画作成・dig への引き継ぎ。refactor 自身はファイル変更、実装、実装オーケストレーションを行わない。

## 対象

$ARGUMENTS

## ハーネス判定

正本は devkit リポジトリの `AGENTS.md`「スキル共通契約」。この SKILL.md は Claude 親 / Codex 親の二層構成で実行する。要点:

- `AskUserQuestion` が使える -> Claude 親。
- なければ `spawn_agent` が使える -> Codex 親。
- どちらでもない -> 判定不能として扱い、選択肢を箇条書きで提示して自由文回答を求める。
- `request_user_input` は plan mode 依存で不安定なため、判定キーに使わない(Codex 親 plan mode の質問手段としてのみ使う)。

## タスクリスト連動

正本は devkit リポジトリの `AGENTS.md`「スキル共通契約 > タスクリスト連動」。refactor 開始時に step 1-5 をタスクリストへ登録し、各 step の開始時に `in_progress`、完了時に `completed` へ更新する(Claude 親: TaskCreate / TaskUpdate、利用不可なら省略可。Codex 親: 組み込み plan 機能または通常の進捗報告で同等の進捗提示)。

## read-only 契約

- allowed-tools に Write / Edit を含めない。ファイル作成・編集・削除・移動を行わない。
- Bash は `git log --numstat`、`git ls-files`、`git blame`、`rg` 相当の読み取り用途だけに使う。生成物の書き込み、format、lint、test 実行、依存更新、コード生成には使わない。
- 挙動変更を伴う案はリファクタリングではなく機能変更として分けて提示する。

## フロー

### 1. スコープ確認

最初に選択肢付き質問で、調査範囲と優先軸を確定する。1 ラウンド最大 4 問とし、回答不足があれば追加で確認する。

- 対象範囲: リポジトリ全体 / 指定ディレクトリ / 指定機能 / 最近変更された範囲
- 動機: 保守性 / 変更容易性 / バグ削減 / テスト容易性 / 性能 / 移行準備
- 後方互換要否: 必須 / 可能なら維持 / 不要 / 不明なのでリスク別に分ける
- 非対象: 触らない領域、今は扱わないリスク、実装しない項目

ハーネス別の質問手段:

- Claude 親: AskUserQuestion を使う。
- Codex 親 plan mode: `request_user_input` を使う。
- Codex 親通常 mode / 判定不能: 選択肢を箇条書きで提示して自由文回答を求める。

締めに、目的 / 成功条件 / 非対象 / 採用した仮定を短く提示して認識を合わせる。

### 2. 棚卸し

親が read-only で調査する。調査対象ごとに、必ず根拠、影響、コスト、リスクを付ける。根拠はできる限り `file:line` 形式で示し、行番号を取れない統計値はコマンド名と集計条件を明記する。

- ホットスポット分析: `git log --numstat` で変更頻度 x サイズを見て、変更量が大きく事故りやすいファイルを抽出する。
- コード重複: 同型処理、似た分岐、近い責務の関数・クラスを `rg` や目視で確認する。
- TODO/FIXME: `TODO` / `FIXME` / `XXX` / `HACK` を探し、放置理由と実装影響を分類する。
- テスト欠落: 重要ロジックに対応するテスト、回帰検出、境界条件テストの有無を確認する。
- 依存の問題: 使われていない依存、責務に対して重すぎる依存、循環・片方向性の崩れ、更新しづらい外部結合を確認する。

棚卸しの出力形式:

```markdown
### 候補: <短い名前>
- 根拠: `path/to/file.ext:123` ...
- 影響: ...
- コスト: ...
- リスク: ...
- 区分: リファクタリング / 機能変更 / 要調査
```

### 3. 優先順位付け

棚卸し結果を、影響 x コスト x リスクで S/A/B/C に格付けする。影響が高く、コストとリスクが低いものを上位に置く。

| ランク | 判断基準 |
|--------|----------|
| S | 影響が高く、手順が小さく、挙動不変を検証しやすい |
| A | 影響が明確で、通常の実装計画に落とせる |
| B | 価値はあるが、事前調査・テスト追加・分割が必要 |
| C | 今回は扱わない。機能変更、破壊的変更、または費用対効果が低い |

格付け後、S/A/B の候補を候補単位の複数選択で提示し、dig へ渡す対象をユーザーが個別に選ぶ。C はレポート記載のみとし選択肢に含めない。

- 区分「要調査」の候補も選択肢に含め、description に「要調査」と明記する。選ばれた要調査候補は「調査を先行し、結果を計画化する」計画草案として dig へ渡す。
- 選択肢の形式: label = 「<ランク>: <短い名前>」、description = 影響・コスト・リスクの要約(1〜2 文。要調査候補はその旨も明記)。
- Claude 親: AskUserQuestion を `multiSelect: true` で使う。1 問あたりの選択肢は 2〜4 個のため、候補 5 件以上はランク順(S→A→B)に 4 件ずつグループ化し、端数が 1 件になる場合は最後の 2 グループを 3+2 に組み替えて 1 件だけの質問を作らない(例: 5 件 = 3+2、6 件 = 4+2、9 件 = 4+3+2)。1 呼び出し最大 4 問、超える分は追加ラウンド。
- 候補 0 件は質問せずレポート提示で終了する。候補 1 件は multiSelect ではなく「引き継ぐ / 見送る」の単一選択にする。
- 各質問の逃げ道は「このグループからは選ばない」に固定する(Other の用途)。全ラウンドの回答を集約した選択合計が 0 件の場合だけレポート提示で終了する。途中の質問で全体方針「レポートのみ」を回答させる形は作らない。
- Codex 親 plan mode: `request_user_input` は multiSelect 非対応・選択肢 2〜3 個のため、候補 1 件につき「引き継ぐ / 見送る」の単一選択質問とし、1 呼び出し最大 3 問で分割する。
- Codex 親通常 mode / 判定不能: 候補一覧を番号付き箇条書き(番号・ランク・名前・要約)で提示し、引き継ぐ候補の番号を自由文で回答してもらう(複数番号可・「なし」可)。
- 選択された候補だけを step 4 の計画化・step 5 の dig 引き継ぎ対象とする。

### 4. リファクタリング計画

選択された候補ごとに、dig へ渡せる計画草案を作る。要調査候補の草案は調査を先行する形で作る。計画は「小さく可逆」「挙動不変」「検証可能」を満たす粒度に分ける。

各項目に必ず含める:

- 目的: 何を改善するか、何は変えないか
- write_scope: 変更候補ファイルの一覧
- 小さく可逆な手順: 1 手順ずつ戻せる単位に分割する
- 挙動不変の検証方法: 既存テスト、追加すべきテスト、手動確認、差分確認
- 完了条件: どの状態になれば dig 側の実装が終わりか
- 非対象: 今回あえて触らない関連負債

### 5. dig への引き継ぎで終了

承認された計画を、dig の step 2 (調査 + 計画) に渡す計画草案として整形して出力する。詳細な実行契約は `plugins/devkit/skills/dig/SKILL.md` を参照する。refactor 自身はここで終了し、実装・backend 選択・レビュー・修正ループはすべて dig 側のフローで行う。

引き継ぎ形式:

```markdown
## dig step 2 計画草案

### 目的
...

### write_scope
- ...

### 実装手順
1. ...

### 検証
- ...

### 非対象
- ...

### refactor 由来の根拠
- `path/to/file.ext:123` ...
```

ハーネス別の終了動作:

- Claude 親: Skill ツールが使える場合は dig を起動し、上記の計画草案を渡す。利用不可なら、ユーザーに `/dig` で計画草案を実行するよう案内する。
- Codex 親: `$dig` を起動し、計画草案を渡すよう案内する。
- どちらの場合も、ユーザーが望むなら計画レポートのみで終了してよい。

## 注意

- 本スキル自身はファイルを変更しない。
- 本スキル自身は実装 backend を選ばない。backend 選択は dig 側の承認フローで行う。
- 挙動変更、API 変更、データ移行、UI 仕様変更、互換性破壊を含む案は、リファクタリング候補と混ぜずに機能変更として提示する。
- 根拠が薄い推測は「要調査」として分け、実装可能な計画に含めない。
